# 결제 운영 진단 Agent 구현 스텝

> 이 문서는 `docs/결제운영진단_Agent_상세설계_v2.md` 기준으로 작성된 구현 가이드다.
> 각 단계는 순서대로 진행해야 하며, 이전 단계가 완료된 상태에서 다음 단계를 시작한다.

---

## 전제 조건

- 작업 브랜치: `dev-agent` (dev-kafka 기반)
- 기존 서비스와 동일한 PostgreSQL DB 공유 (`skala-postgres` 컨테이너), `incident` 스키마 신규 추가
- 기술 스택: Java 17 + Spring Boot 3.x (기존 서비스와 동일)
- LLM 클라이언트: Anthropic Java SDK (`claude-sonnet-4-6` 사용)
- 새 서비스 3개: `incident-detector`, `incident-agent`, `incident-api`

## Phase 1. DB 스키마 정의

**목적**: 세 서비스가 공유하는 incident 스토어 테이블 생성

### Step 1-1. SQL 스키마 파일 작성

**파일**: `infra/postgres/incident_schema.sql`

생성할 테이블:

```sql
CREATE SCHEMA IF NOT EXISTS incident;

-- 탐지된 사건 원장
CREATE TABLE incident.incidents (
    incident_id        UUID         PRIMARY KEY,
    incident_type      VARCHAR(50)  NOT NULL,  -- DUPLICATE_PAYMENT | GHOST_ORDER | ZOMBIE_HOLD | UNCONFIRMED_PAYMENT
    incident_key       VARCHAR(255) NOT NULL,  -- bookingId 또는 paymentId
    status             VARCHAR(30)  NOT NULL DEFAULT 'OPEN',
    severity           VARCHAR(20)  NOT NULL,  -- critical | high | medium | low
    confidence         NUMERIC(5,4),
    primary_payment_id UUID,
    primary_booking_id UUID,
    user_id            BIGINT,
    concert_id         BIGINT,
    schedule_id        BIGINT,
    first_detected_at  TIMESTAMPTZ  NOT NULL,
    last_detected_at   TIMESTAMPTZ  NOT NULL,
    last_analyzed_at   TIMESTAMPTZ,
    latest_analysis_version INT     DEFAULT 0,
    needs_human_approval    BOOLEAN DEFAULT FALSE,
    current_state_jsonb     JSONB,
    open_reason_signal      VARCHAR(100),
    resolved_at        TIMESTAMPTZ,
    resolved_by        VARCHAR(100),
    created_at         TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at         TIMESTAMPTZ  NOT NULL DEFAULT now()
);

-- 같은 type+key 쌍의 OPEN incident는 중복 생성 금지
CREATE UNIQUE INDEX uq_incident_open
    ON incident.incidents (incident_type, incident_key)
    WHERE status NOT IN ('RESOLVED');

-- LLM 분석 이력 (버전 관리)
CREATE TABLE incident.incident_analysis_versions (
    analysis_version_id UUID        PRIMARY KEY,
    incident_id         UUID        NOT NULL REFERENCES incident.incidents(incident_id),
    version_number      INT         NOT NULL,
    analysis_status     VARCHAR(30) NOT NULL,  -- PENDING | RUNNING | COMPLETED | FAILED | PENDING_RETRY
    model_provider      VARCHAR(50),
    model_name          VARCHAR(100),
    input_schema_version  VARCHAR(30),
    output_schema_version VARCHAR(30),
    analysis_input_jsonb  JSONB,
    analysis_output_jsonb JSONB,
    summary_text        TEXT,
    error_code          VARCHAR(50),
    error_message       TEXT,
    requested_by        VARCHAR(100),
    trigger_type        VARCHAR(50),  -- NEW_INCIDENT | STATE_CHANGED | MANUAL_REANALYZE | RETRY
    created_at          TIMESTAMPTZ  NOT NULL DEFAULT now(),
    completed_at        TIMESTAMPTZ,
    UNIQUE (incident_id, version_number)
);

-- incident 상태 변경 이력
CREATE TABLE incident.incident_status_history (
    history_id    UUID        PRIMARY KEY,
    incident_id   UUID        NOT NULL REFERENCES incident.incidents(incident_id),
    from_status   VARCHAR(30),
    to_status     VARCHAR(30) NOT NULL,
    changed_by    VARCHAR(100),
    change_reason TEXT,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- incident-detector Kafka 이벤트 중복 제거 (Inbox)
CREATE TABLE incident.detector_inbox_events (
    id              UUID         PRIMARY KEY,
    dedupe_key      VARCHAR(255) NOT NULL UNIQUE,  -- "kafka:{eventId}"
    event_type      VARCHAR(100),
    source_topic    VARCHAR(100),
    received_at     TIMESTAMPTZ  NOT NULL DEFAULT now(),
    duplicate_count INT          DEFAULT 0
);

-- 유령 주문 탐지를 위한 교차 토픽 상태 추적
-- PAYMENT_CONFIRMED 수신 후 booking.confirmed 미수신 여부 추적
CREATE TABLE incident.detector_pending_correlations (
    id                 UUID         PRIMARY KEY,
    correlation_type   VARCHAR(50)  NOT NULL,  -- WAITING_BOOKING_CONFIRM | WAITING_PAYMENT_CONFIRM
    key_type           VARCHAR(30)  NOT NULL,  -- BOOKING_ID | PAYMENT_ID
    key_value          VARCHAR(255) NOT NULL,
    trigger_event_type VARCHAR(100),
    triggered_at       TIMESTAMPTZ  NOT NULL,
    deadline_at        TIMESTAMPTZ  NOT NULL,  -- 이 시각까지 응답 없으면 incident 생성
    extra_jsonb        JSONB,                  -- paymentId, userId, concertId 등 부가 정보
    resolved           BOOLEAN      DEFAULT FALSE,
    created_at         TIMESTAMPTZ  NOT NULL DEFAULT now(),
    UNIQUE (correlation_type, key_type, key_value)
);

-- 좀비 예약 탐지를 위한 booking 종료 추적 (Redis 폴링 대상)
CREATE TABLE incident.detector_zombie_candidates (
    id               UUID         PRIMARY KEY,
    booking_id       UUID         NOT NULL UNIQUE,
    user_id          BIGINT,
    concert_id       BIGINT,
    schedule_id      BIGINT,
    ended_event_type VARCHAR(50),             -- booking.canceled | booking.expired
    ended_at         TIMESTAMPTZ  NOT NULL,
    check_after_at   TIMESTAMPTZ  NOT NULL,  -- ended_at + grace_period (기본 1분)
    checked          BOOLEAN      DEFAULT FALSE,
    created_at       TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

- `infra/postgres/init.sql`에 `\i incident_schema.sql` 또는 직접 append
- PostgreSQL 컨테이너 재시작 시 자동 반영되도록 volume mount 확인

```java
// dedupe_key unique 제약으로 DB 레벨에서 중복 차단
@Transactional
public boolean tryRecord(String dedupeKey, String eventType, String sourceTopic) {
    try {
        DetectorInboxEvent ev = new DetectorInboxEvent();
        ev.setId(UUID.randomUUID());
        ev.setDedupeKey(dedupeKey);  // "kafka:{eventId}"
        ev.setEventType(eventType);
        ev.setSourceTopic(sourceTopic);
        ev.setReceivedAt(OffsetDateTime.now(ZoneOffset.UTC));
        ev.setDuplicateCount(0);
        repository.save(ev);
        return true;
    } catch (DataIntegrityViolationException e) {
        // unique 제약 위반 → 중복
        repository.incrementDuplicateCount(dedupeKey);
        return false;
    }
}
```

### Step 2-4. 4가지 탐지 룰 구현

#### Rule 1: 중복 결제 (`DuplicatePaymentRule`)

```
트리거: PAYMENT_CONFIRMED 수신 시
로직:
  - incidents 테이블에서 동일 bookingId + type=DUPLICATE_PAYMENT + status != RESOLVED 조회
  - 없으면: 동일 bookingId로 PAYMENT_CONFIRMED 이벤트가 이미 inbox에 있는지 확인
  - 두 번째 PAYMENT_CONFIRMED 감지 시 → createIncident(DUPLICATE_PAYMENT, severity=critical)
  - 이미 incident 있으면 → updateIncident(last_detected_at, current_state)
```

#### Rule 2: 유령 주문 (`GhostOrderRule`)

```
트리거 A (PAYMENT_CONFIRMED 수신):
  - detector_pending_correlations에 INSERT
    (type=WAITING_BOOKING_CONFIRM, key=BOOKING_ID, value={bookingId})
    deadline_at = now + ${detector.ghost-order.threshold-minutes:5}분
    extra_jsonb = {paymentId, userId, concertId, scheduleId}

트리거 B (booking.confirmed 수신):
  - detector_pending_correlations에서 동일 bookingId, type=WAITING_BOOKING_CONFIRM 조회
  - 존재하면 resolved=true 업데이트 → 정상 처리, incident 없음

스케줄러 (PendingCorrelationCheckScheduler, 30초 주기):
  - deadline_at < now AND resolved=false 인 row 조회
  - 각 row에 대해 createIncident(GHOST_ORDER, severity=high)
  - row resolved=true 처리
```

#### Rule 3: 미확정 결제 (`UnconfirmedPaymentRule`)

```
트리거 A (PAYMENT_PAID 수신):
  - detector_pending_correlations에 INSERT
    (type=WAITING_PAYMENT_CONFIRM, key=PAYMENT_ID, value={paymentId})
    deadline_at = now + ${detector.unconfirmed.paid-threshold-minutes:2}분

트리거 B (WEBHOOK_* 수신 — eventType이 "WEBHOOK_"로 시작):
  - 동일한 방식으로 pending 등록 (이미 있으면 skip)

트리거 C (PAYMENT_CONFIRMED 수신):
  - 동일 paymentId의 WAITING_PAYMENT_CONFIRM pending을 resolved=true 처리

스케줄러:
  - deadline 초과한 WAITING_PAYMENT_CONFIRM 조회 → createIncident(UNCONFIRMED_PAYMENT, severity=high)
```

#### Rule 4: 좀비 예약 (`ZombieHoldRule` + `ZombieHoldCheckScheduler`)

```
트리거 (booking.canceled / booking.expired 수신):
  - detector_zombie_candidates에 INSERT
    (bookingId, userId, concertId, scheduleId, endedAt=now)
    check_after_at = now + ${detector.zombie.grace-period-seconds:60}초

스케줄러 (ZombieHoldCheckScheduler, 30초 주기):
  - check_after_at < now AND checked=false 인 row 조회
  - 각 row에 대해 ZombieHoldChecker.hasHoldRemaining(userId, concertId, scheduleId) 호출
    → Redis에서 seatAccessKey / seatLockOwner 조회
  - hold 잔존 시 → createIncident(ZOMBIE_HOLD, severity=medium)
  - checked=true 처리
```

**파일**: `zombie/ZombieHoldChecker.java`

```java
// Redis 키 패턴은 ticketing-service의 RedisKeyGenerator와 동일해야 함
// "seat:access:{userId}:{concertId}:{scheduleId}" 형태
public boolean hasHoldRemaining(Long userId, Long concertId, Long scheduleId) {
    String accessKey = "seat:access:" + userId + ":" + concertId + ":" + scheduleId;
    return Boolean.TRUE.equals(redisTemplate.hasKey(accessKey));
}
```

**유의사항**: Redis 키 패턴이 ticketing-service의 `RedisKeyGenerator`와 반드시 일치해야 한다. 키 구조가 변경되면 여기도 같이 변경해야 한다.

### Step 3-2. AnalysisJobScheduler 구현

```
@Scheduled(fixedDelayString = "${agent.poll.fixed-delay-ms:10000}")

public void pollAndAnalyze() {

public LlmResponse call(String systemPrompt, String userMessage) {

    Message response = client.messages().create(
        MessageCreateParams.builder()
            .model(model)  // ${agent.llm.model:claude-sonnet-4-6}
            .maxTokens(1024)
            .system(systemPrompt)
            .addUserMessage(userMessage)
            .build()
    );
    return new LlmResponse(response.content().get(0).text());
}
```

**timeout 설정**: 기본 10초 (설계 19.1절)
**retry 횟수**: 최대 2회 (설계 19.2절)

## 주요 구현 원칙 (반드시 지킬 것)

1. **Redis 키 패턴**: `ZombieHoldChecker`의 키 구조는 ticketing-service `RedisKeyGenerator`와 완전히 동일해야 한다. 키가 바뀌면 좀비 예약 탐지가 무음으로 실패한다.
2. **Inbox dedupe_key 형식**: `"kafka:{eventId}"` — eventId는 Kafka 메시지의 `eventId` 필드 (UUID).
3. **incident unique 제약**: 동일 `(incident_type, incident_key)` 쌍에 OPEN incident가 이미 있으면 신규 생성이 아닌 `last_detected_at` 갱신.
4. **LLM 출력 PII 검사**: `AnalysisOutputValidator`에서 summary, recommendedActions에 이메일/전화번호 패턴이 포함되면 해당 버전을 FAILED 처리.
5. **모든 임계값은 config화**: 탐지 시간 기준, severity 보정 기준, 폴링 주기 모두 `application.properties` + 환경변수로 외부화.
6. **트랜잭션 경계**: Kafka Consumer 메서드는 `@Transactional`. Inbox 기록과 incident 생성이 같은 트랜잭션에서 처리되어야 한다.
