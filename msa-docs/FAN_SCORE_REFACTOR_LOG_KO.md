# 팬점수 리팩토링 작업 기록

## 배경

- 목표: 사용자가 특정 콘서트 대기열에 진입할 때, 해당 콘서트의 아티스트에 대한 팬점수가 높은 사용자에게 시간 기반 우선순위를 부여한다.
- 기존 코드는 전역 값인 `users.fan_score` 하나만 사용하고 있었지만, 실제 요구사항은 아티스트별 팬점수이다.
- 현재 서비스 내부에서 신뢰 가능하게 팬점수에 반영할 수 있는 데이터는 같은 아티스트의 확정 예매 이력이다.
- 외부 팬클럽 인증은 이번 범위에서 제외하지만, 이후 확장 가능하도록 구조는 열어둔다.

## 기존 상태 확인

- `QueueService`는 기존에 `users.fan_score`를 읽어 대기열 가중치에 사용하고 있었다.
- `concerts.artist_id`는 이미 존재한다.
- 예매 이력은 아래 경로로 아티스트까지 추적 가능하다.
  - `bookings.schedule_id`
  - `schedules.concert_id`
  - `concerts.artist_id`
- 현재 DB/모델에는 아티스트별 팬점수 저장 구조가 없었다.

## 계획한 작업 방향

1. 아티스트별 팬점수 저장 구조를 추가한다.
2. 대기열 로직이 전역 점수가 아닌 `user + artist` 기준 점수를 사용하도록 바꾼다.
3. 같은 아티스트의 확정 예매 이력으로부터 점수를 계산한다.
4. 이후 외부 팬클럽 점수 반영이 가능하도록 구조를 열어둔다.
5. 구현 내용과 점수 규칙을 문서로 정리한다.

## 작업 순서

1. 아티스트별 팬점수용 DB 구조 추가
2. 아티스트별 점수 조회/갱신용 repository/service 추가
3. 기존 확정 예매 이력 기반 백필 처리 추가
4. 콘서트의 아티스트를 찾아 해당 점수를 대기열에 반영하도록 수정
5. `/api/users/me` 및 프론트 표시 의미 정리
6. 빌드 및 동작 검증

## 실제 반영 내용

- 신규 DB 요약 모델 `user_artist_fan_scores` 추가
  - 키: `user_id + artist_id`
  - 필드: `booking_score`, `external_score`, `total_score`, `created_at`, `updated_at`
- 대기열 우선순위 조회 로직을 아래 기준으로 변경
  - 대상 `concertId`
  - 해당 공연의 `artist_id`
  - `user_id + artist_id` 기준 아티스트별 팬점수
- 기존 확정 예매 이력을 기준으로 booking 기반 팬점수를 채우는 시작 시점 동기화 로직 추가
- 결제 확정 시 해당 아티스트 팬점수가 즉시 증가하도록 연결
- `/api/users/me`의 `fanScore`는 사용자 전체 아티스트 점수 총합을 반환하도록 변경
- `MyPage` 문구를 수정하여 “전역 점수 = 모든 공연에서의 대기열 우선순위”처럼 오해되지 않도록 정리

## 이번 리팩토링의 점수 규칙

- 현재 내부적으로 신뢰 가능한 아티스트 팬점수 소스:
  - 같은 아티스트의 확정 예매 이력
- 점수 적립 단위:
  - 같은 아티스트의 `CONFIRMED` 예매 1건 = `1000점`
- 대기열 우선순위 혜택:
  - 아티스트별 팬점수를 시간 우선순위로 변환
  - 최대 `5000ms`까지 반영

## 전제 및 가정

- 현재 서비스에는 실제 관람 완료 여부 데이터가 없기 때문에, 내부적으로 확인 가능한 최선의 대체 지표로 `CONFIRMED` 예매 이력을 사용한다.
- 점수 적립 기준은 좌석 수가 아니라 예매 건수 기준이다.
- 외부 팬클럽 인증은 아직 구현하지 않았지만, 이후 반영할 수 있도록 `external_score` 필드를 추가했다.

## 검증 결과

- 백엔드 컴파일 검증:
  - `./gradlew compileJava` 성공
- 프론트 빌드 검증:
  - 현재 실행 환경에 `npm`이 없어 수행하지 못함

## 참고 메모

- 권장 기본 적립 규칙:
  - 같은 아티스트의 확정 예매 1건 = 1회 적립
- 대기열 혜택은 계속 시간 기반으로 유지한다.
- 레거시 `users.fan_score` 컬럼은 코드상 핵심 대기열 로직에서는 더 이상 사용하지 않는다.
