## 현재 메모

- 현재 코드는 실행 단위는 분리돼 있지만 `shared-kernel` 과 공용 DB 관성 때문에 서비스 경계가 강하게 닫혀 있지 않다.
- 특히 `payment-service` 가 booking/seat 에 직접 관여하는 부분이 가장 중요한 1차 수정 대상이다.
- 1차 감사 결과:
  - `payment-service/PaymentService#confirmPayment` 가 좌석 reserve, booking confirmed, fan score 적용, Redis hold cleanup 까지 직접 수행한다.
  - `payment-service/PaymentScheduler` 가 booking cancel 과 seat hold release 를 직접 수행한다.
  - `payment-service` 는 booking/seat shared repository 와 entity 에 직접 의존한다.
  - `ticketing-service` 는 booking 생성/hold 까지는 명확하지만 finalization/reconciliation owner 구조는 아직 없다.
  - `queue-service`, `ticketing-service` 는 `user-auth-service` 내부 API 를 HTTP 로 직접 호출한다.
  - 여러 서비스 설정 파일에 `your-secret-key...`, `localhost:5173`, 고정 `@Scheduled(fixedDelay=...)` 같은 하드코딩/약한 기본값이 남아 있다.
  - 현재 `payment-service` 에는 `/api/payments/ops/summary` 정도만 있고 설계 문서 기준의 운영 API 묶음은 없다.
- 첫 구현 배치 가정:
  - `Kafka/outbox` 실제 도입은 다음 배치로 미룬다.
  - 현재 배치는 `boundary separation + reconciliation scaffolding` 을 우선한다.
  - public payment API 는 유지하고 내부 finalization ownership 만 먼저 이동한다.
- Batch 1 반영 결과:
  - `ticketing-service` 에 `/internal/finalizations` 내부 API 와 finalization owner 서비스가 추가됐다.
  - `payment-service` 의 confirm/cancel/expire 경로는 booking/seat/Redis 직접 갱신 대신 internal finalization client 를 사용한다.
  - `payment-service` 의 hold validity 사전 판단은 제거되고, hold 판정 책임은 `ticketing-service` 로 이동했다.
  - `PaymentScheduler` 도 ticketing internal finalization 을 사용하도록 이동했다.
  - `ticketing-service` 애플리케이션 스캔 범위에 `modules.finalization`, `modules.fanscore` 를 포함시켰다.
  - redirect URL 은 설정값으로 외부화했다.
- Batch 2 반영 결과:
  - `user-auth-service` 의 `/internal/users/**` 는 `X-Internal-Api-Key` 기반 내부 인증을 요구한다.
  - `queue-service`, `ticketing-service` 는 내부 사용자 조회 호출 시 같은 헤더를 자동 전파한다.
  - active 서비스들에 `spring-boot-starter-actuator` 와 `health/info`, `liveness/readiness` baseline 이 추가됐다.
  - 서비스별 보안 설정에서 `/actuator/health`, `/actuator/health/**`, `/actuator/info` 만 공개하고 나머지 인증 규칙은 유지했다.
  - `payment-service` 의 `ops/summary` 는 일반 사용자 JWT 외에 별도 운영 토큰 헤더를 요구하도록 강화 중이다.
