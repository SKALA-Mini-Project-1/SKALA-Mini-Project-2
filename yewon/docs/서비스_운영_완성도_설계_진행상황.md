- 기준 코드: `dev` 브랜치에서 분기한 `dev-refactoring`
- 기준 설계 문서: 현재 사용자 브랜치의 `서비스_운영_완성도_설계.md`
- 작업 원칙:
  - 한 번에 전체 수정하지 않는다.
  - 배치 단위로 수정하고 검증한다.

  - 병렬 처리 가능한 부분은 서브 에이전트를 활용한다.

## 현재 메모

- 현재 코드는 실행 단위는 분리돼 있지만 `shared-kernel` 과 공용 DB 관성 때문에 서비스 경계가 강하게 닫혀 있지 않다.
- 특히 `payment-service` 가 booking/seat 에 직접 관여하는 부분이 가장 중요한 1차 수정 대상이다.
- 1차 감사 결과:
  - `payment-service/PaymentService#confirmPayment` 가 좌석 reserve, booking confirmed, fan score 적용, Redis hold cleanup 까지 직접 수행한다.
  - `payment-service/PaymentScheduler` 가 booking cancel 과 seat hold release 를 직접 수행한다.
  - `payment-service` 는 booking/seat shared repository 와 entity 에 직접 의존한다.
  - `ticketing-service` 는 booking 생성/hold 까지는 명확하지만 finalization/reconciliation owner 구조는 아직 없다.
  - `queue-service`, `ticketing-service` 는 `user-auth-service` 내부 API 를 HTTP 로 직접 호출한다.
  - 여러 서비스 설정 파일에 `your-secret-key...`, `localhost:5173`, 고정 `@Scheduled(fixedDelay=...)` 같은 하드코딩/약한 기본값이 남아 있다.
  - 현재 `payment-service` 에는 `/api/payments/ops/summary` 정도만 있고 설계 문서 기준의 운영 API 묶음은 없다.
- 첫 구현 배치 가정:
  - `Kafka/outbox` 실제 도입은 다음 배치로 미룬다.
  - 현재 배치는 `boundary separation + reconciliation scaffolding` 을 우선한다.
  - public payment API 는 유지하고 내부 finalization ownership 만 먼저 이동한다.
- Batch 1 반영 결과:
  - `ticketing-service` 에 `/internal/finalizations` 내부 API 와 finalization owner 서비스가 추가됐다.
  - `payment-service` 의 confirm/cancel/expire 경로는 booking/seat/Redis 직접 갱신 대신 internal finalization client 를 사용한다.
  - `payment-service` 의 hold validity 사전 판단은 제거되고, hold 판정 책임은 `ticketing-service` 로 이동했다.
  - `PaymentScheduler` 도 ticketing internal finalization 을 사용하도록 이동했다.
  - `ticketing-service` 애플리케이션 스캔 범위에 `modules.finalization`, `modules.fanscore` 를 포함시켰다.
  - redirect URL 은 설정값으로 외부화했다.
- Batch 2 반영 결과:
  - `user-auth-service` 의 `/internal/users/**` 는 `X-Internal-Api-Key` 기반 내부 인증을 요구한다.
  - `queue-service`, `ticketing-service` 는 내부 사용자 조회 호출 시 같은 헤더를 자동 전파한다.
  - active 서비스들에 `spring-boot-starter-actuator` 와 `health/info`, `liveness/readiness` baseline 이 추가됐다.
  - 서비스별 보안 설정에서 `/actuator/health`, `/actuator/health/**`, `/actuator/info` 만 공개하고 나머지 인증 규칙은 유지했다.
  - `payment-service` 의 `ops/summary` 는 일반 사용자 JWT 외에 별도 운영 토큰 헤더를 요구하도록 강화 중이다.
  - `payment-service`, `queue-service` 의 scheduler delay 와 payment expiration 정책 일부를 설정값으로 외부화했다.
- Batch 2.5 진행 중:
  - `payment-service` 의 `createPayment`, `ensureOwner` 가 ticketing 내부 조회 API 를 사용하도록 read-side decoupling 1차를 진행 중이다.
  - 결제 생성/권한 판정에 필요한 booking context 는 ticketing 내부 조회 API 뒤로 이동했다.
  - `payment-service` 의 payment history 상세 렌더링은 ticketing batched internal read 로 이동해 booking/seat 직접 조회를 제거했다.
  - `payment-service` 의 user-scoped history/refund 조회는 ticketing 의 `user -> bookingIds` 내부 조회를 거쳐 `payments` 테이블만 읽도록 정리했다.
  - `paymentRepository` 의 `payments JOIN bookings` 네이티브 조회는 제거됐다.
  - `queue-service` 의 schedule 검증은 direct DB 조회 대신 `concert-service` 내부 조회 API 로 이동했다.
  - `queue-service` 의 fan score 우선순위 계산은 shared `FanScoreService` 직접 호출 대신 queue 전용 `QueuePriorityService` 로 이동했다.
  - `queue-service` 애플리케이션 스캔에서 불필요한 `modules.users` 범위를 제거했다.
  - `payment-service` 의 이벤트 기록은 `PaymentEventRecorder` 로 1차 통합해 service/scheduler 간 중복 생성 로직을 줄였다.
  - 결제 생성 시점에도 `PAYMENT_CREATED` 이벤트가 남도록 보강했다.
  - `payment_events` 는 `eventId`, `eventVersion`, `producer`, `aggregateType`, `aggregateId`, `orderingKey`, `correlationId`, `causationId`, `traceId`, `occurredAt` 필드를 갖는 canonical envelope 방향으로 확장했다.
  - 아직 broker publish 나 ticketing inbox 소비는 붙이지 않았고, 현재 단계는 persistent event shape 정규화까지 진행했다.
  - `ticketing-service` 에 `reconciliation_tasks` 최소 엔티티/레포지토리/서비스를 추가했다.
  - `TicketingFinalizationService` 는 `INVALID_HOLD`, `BOOKING_ALREADY_CANCELED`, `BOOKING_ALREADY_CONFIRMED` 결과를 reconciliation backlog 로 적재한다.
  - 같은 mismatch 조합의 중복 backlog 적재를 막는 dedupe 체크를 추가했다.
  - `ticketing-service` 에 reconciliation backlog monitor scheduler 를 추가해 `REQUESTED` task 누적 시 경고 로그를 남기도록 했다.
  - `ticketing-service` 에 internal ops reconciliation summary/list 조회 API 를 추가해 backlog 상태를 바로 확인할 수 있게 했다.
  - `ticketing-service` 에 reconciliation replay scheduler 를 추가해 `BOOKING_ALREADY_CONFIRMED` 는 자동 완료하고, 나머지 mismatch 는 `WAITING_MANUAL_APPROVAL` 로 승격한다.
  - `ticketing-service` 에 internal ops retry endpoint 를 추가해 `WAITING_MANUAL_APPROVAL` task 를 다시 `REQUESTED` 로 되돌릴 수 있게 했다.
  - 현재 단계는 mismatch backlog 적재 + dedupe + monitor + ops 조회 + 최소 replay + 수동 retry 트리거까지 진행했고, inbox dedupe store 는 아직 남아 있다.
