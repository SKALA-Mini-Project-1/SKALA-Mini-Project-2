## 현재 확인된 갭

- 실제 runtime entrypoint 인 `backend` 에는 설계 문서 수준의 `reconciliation ops API` 가 없다.
- 현재 검증 가능한 운영 API 는 실질적으로 `GET /api/payments/ops/summary` 수준이다.
- `ticketing-service` 기준으로 설계했던 `reconciliation/inbox` 운영 경로는 현재 runtime path 에 직접 연결되어 있지 않다.

## 현재 판단

- 현재 worktree 에서 실제 기동/핵심 사용자 흐름은 `backend` 기준으로 동작한다.
- 설계 문서 기준 구조 리팩토링은 별도 서비스 디렉터리 방향으로 진행됐지만, 실제 런타임 엔트리포인트는 아직 `backend` 다.
- 따라서 `backend` 제거는 아직 바로 할 수 없고, 서비스별 독립 엔트리포인트를 닫은 뒤 마지막 단계에서 삭제해야 한다.
