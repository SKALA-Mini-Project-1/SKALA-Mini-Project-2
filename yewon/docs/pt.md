제 기준에서 고도화 방향은 크게 4개로 정리됩니다.

운영 안정화
민감정보 분리, 환경별 설정 분리, 예외 응답 정리, 실행/배포 재현성 확보.

품질 기반 만들기
대기열, 좌석 선점, 결제 상태전이 중심으로 테스트 추가. 발표자료의 강점을 실제 신뢰도로 바꾸는 단계예요.

구조 리팩터링
결제 서비스 분해, 프론트 라우팅/상태관리 정리, 도메인 경계 명확화.

제품 고도화
팬 점수 정책 정교화, 운영 지표 대시보드, 알림/이벤트 아키텍처, 관리자 기능.

## Agent 자료

### 에이전트 소개
**결제 운영 진단 Agent**
**이름 정하기**

결제, 예약, 좌석 상태의 불일치를 incident 단위로 묶고, AI가 원인과 다음 조치를 설명해 운영자의 빠른 판단을 돕는 운영 보조 시스템

발표용 한 줄 문장:
결제 운영 진단 Agent는 복잡한 결제 이상 상황을 사람이 이해하기 쉬운 사건 단위로 정리하고, 원인 추정과 대응 방향을 제안하는 AI 기반 운영 보조 계층입니다.

## 왜 필요한가

결제 문제는 단순 성공/실패만으로 설명되지 않음, PG 승인과 내부 상태가 어긋날 수 있음
booking, seat, webhook, Redis 상태를 함께 봐야 원인 파악 가능

-> 운영자가 여러 로그와 테이블을 직접 대조해야 하는 비효율 존재
-> 복잡한 결제 이상 상황을 사람이 이해하기 쉬운 사건 단위로 정리 -> 원인 추정과 대응 방향을 제안 -> 운영자의 빠른 판단을 돕는다.

장표 문장 예시:
결제 장애는 하나의 로그만으로 설명되지 않습니다. PG, Payment, Booking, Seat, Redis 상태를 함께 비교해야만 실제 원인을 파악할 수 있습니다.

## 에이전트의 핵심 역할

탐지 : 이상 상황을 incident로 정리
해석 : 사건의 가능 원인을 설명
조치 제안 : 운영자에게 다음 액션을 제안

한 줄 정리:
단순 경고를 넘어서, 무엇이 문제인지와 어디를 먼저 봐야 하는지까지 설명합니다.

## 다루는 주요 incident 유형

중복 결제: 같은 booking 또는 좌석에 결제가 중복 반영된 상황
유령 주문: 결제는 성공했지만 booking/seat 후처리가 완료되지 않은 상황
좀비 예약: 사용자가 이탈했지만 좌석 hold가 남아 다른 사용자를 막는 상황
미확정 결제: PG 승인 이후에도 내부 결제 확정이 끝나지 않은 상황

## 전체 구조

**흐름도? 추가**

서비스에서 발생한 상태 변화를 Kafka로 수집
detector가 룰 기반으로 이상 incident를 생성
agent가 사건을 해석해 요약, 원인, 권장 액션 생성
api가 운영자 화면에 incident 목록과 상세 정보를 제공

detector: 룰 기반 이상 탐지, incident 생성/갱신
agent: 사건 해석, 자연어 요약, 원인 추정, 조치 제안

발표용 문장:
detector가 이상 여부를 기계적으로 판별한다면, agent는 그 결과를 운영 가능한 정보로 바꿔줍니다.
이 부분은 장표에서 차별점으로 잘 먹힙니다.

## 운영자에게 제공하는 결과

**이미지로 보여주면서 설명**

사건 유형
심각도
영향 범위
현재 상태
원인 추정
근거 타임라인
권장 액션
발표용 예시 문장:

PG 승인 이후 payment는 PAID까지 반영되었으나 booking CONFIRMED 이벤트가 확인되지 않았습니다. 후처리 누락 가능성이 높습니다.

권장 액션 예시:
webhook 재처리 확인
booking 상태 재조회
좌석 hold TTL 및 owner 재확인
refund 필요 여부 검토

## 기대 효과

결제 이상 상황을 더 빠르게 발견
운영자의 원인 분석 시간 단축
로그/테이블 대조 작업 감소
장애 조사 방식 표준화
향후 관리자 툴 및 자동 복구로 확장 가능
발표용 마무리 문장:

결제 운영 진단 Agent는 복잡한 결제 장애를 더 빠르게 발견하고, 더 짧은 시간 안에 더 정확하게 대응할 수 있도록 돕습니다.
