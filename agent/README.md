**실시간 이벤트 기반 에이전트(Event-Driven Agent)**

Agent 역할
: 서버에 문제가 발생했을 때, 왜 죽었는지와 어떻게 해결해야하는지 제안하는 에이전트

에러가 터졌을 때, 어떻게 분석을 해서 에러 원인을 찾을 것인가...
코드가 서버에 없는데,,, 이미지 파일을 가지고 찾는다? 아니면 어떤 서비스에서 오류가 터졌고, ~~ 로그가 발생했을니까 어떤 서비스의 어떤 기능을 찾아봐라

1. log 수집 방식
    
    1.1 Kafka를 메인 입력원(비즈니스 이벤트)으로 사용 (비즈니스 이상 탐지): 티켓 예매 성공/실패, 결제 요청/완료 등의 비즈니스 이벤트를 Kafka 토픽으로 발행 (에이전트는 이 토픽을 구독하다가 '결제 요청 후 5분간 완료 이벤트 없음' 같은 상황이 발생하면 추론을 시작)
    1.2 Prometheus/Loki를 보조 수단으로 활용 (인프라 상태 조회): 에이전트가 Kafka 로그를 보고 "어? 에러가 많네?"라고 판단하면, **도구(Tool Usage)**를 사용해 프로메테우스 API를 호출 (인프라 지표를 가져와서 에러의 원인이 '코드'인지 '인프라'인지 판별)
2. kafka 이벤트를 받을까..?
    
    kafka 이벤트는 (Trace ID, User ID, Timestamp) 이러한 정보만 가지고 있음
    에이전트가 조사가 필요할 로그라고 판단하면 그때 가져온 kafka 이벤트의 trace_id로 전체 흐름을 긁어오도록!
    Trace ID: 여러 서비스를 거칠 때, 모든 서비스의 로그에 동일한 trace_id가 남도록 설정 -> trace_id를 키값으로 사용해 로그 저장소에서 해당 요청의 전체 타임라인을 한꺼번에 가져옴
    이러한 로그 정보들을 별도로 저장하는 곳 필요: 'ELK 스택' or 'PLG 스택'이라는 별도의 로그 분석 인프라를 구축!
    Loki / Elasticsearch에 모든 MS에서 발생하는 로그들을 쌓아 둠. (여기에 trace_id를 기반으로 인덱싱되어 있음 -> 대량의 로그 중 특정 trace_id만 뽑기 가능!) -> Collector(Fluentbit or Logstash 등의 도구)로 각 서비스의 로그를 긁어 Loki, Elasticsearch로 보내줌
3. Agent 구조
    
    1. kafka 이벤트 트래킹
    2. 오류 로그 확인 (Loky)
    3. 이전에 해결했던 방식에 대해 탐색
    4. 해결 방안 정리
    5. 운영자 대시보드
4. 운영 대시보드
    수집하고 분석한 내용을 바탕으로 인프라 관리자에게 전송
    
    예) "결제 중복 에러가 50건 발생했습니다. Kafka 로그 분석 결과 특정 사용자 패턴이 식별되었고, Prometheus 확인 결과 DB 커넥션 풀이 가득 찼습니다. 해당 IP 차단과 커넥션 증설을 권장합니다."

5. 서비스 흐름
Kafka가 에러 발생 이벤트 알림만 Agent한테 던짐 (해당 메시지 내에 trace_id가 포함) -> Agent는 전달 받은 trace_id로 Loki API한테 trace_id로 전후 로그를 가져옴

6. 트래킹할 에러의 종류

- 매크로 탐지: 매크로 동작 탐지 -> 후조치
매크로 잡는게 매력적인 구조일까?
실무적 관점으로 봤을 때, 득이 되는 선택인가?

이상 탐지...

운영자의 관점에서 득이 되는 가치를 줘야한다 (1가지 문제를 타겟하는 에이전트)
에이전트로 만들었을 때, 단순 코드로 개발한 것보다 나은 가치를 줄 수 있는 것엔 뭐가 있을까


- 대량 트래픽 발생 시 문제 발생 시나리오

- 스코어 예측 A~Z 흐름의 스코어

- 결제 단계에서 발생하는 문제를 트래킹하는 Agent
- 중복 결제 발생: 
- 유령 주문: 결제 서비스에서는 '성공'으로 기록되었으나, 주문 서비스에는 '대기' 상태로 남아 티켓 발행이 안 되는 상황
- 좀비 예약: 사용자가 결제창 이탈했음에도 Redis나 DB에 좌석 점유 상태가 해제되지 않아 다른 사용자가 예매하지 못하는 상황
- 미확정 결제: PG사(결제 대행사)에서는 승인이 났으나, 우리 서버로 오는 Webhook(결제 결과 통지)이 유실되어 시스템상에서는 '미결제'로 남아있는 경우

## 1. 로그 필터링 및 컨텍스트 확보 로직
Kafka 메시지 단일이 아닌 원인 파악을 위한 전후 과정을 에이전트에게 제공해야 함
- 샘플링 및 집계 (Windowing): 400, 500 에러가 단발성인지, 특정 API에서 급증하는 중인지 판단하는 로직이 필요. (예: "지난 1분간 특정 좌석 선점 API에서 500 에러가 100건 발생함")
- 관련 로그 추적 (Trace ID): 오류가 터진 시점의 로그뿐만 아니라, 동일한 trace_id를 가진 이전 로그들을 묶어서 에이전트에게 전달함. 그래야 "사용자가 어떤 경로로 들어와서 에러가 났는지"를 알 수 있음.

## 2. 지식 베이스 연동
- 사내 Wiki/Git Issue 연동: 과거에 발생했던 비슷한 장애의 대응 이력이나, 프로젝트의 API 명세서를 RAG(검색 증강 생성) 방식으로 에이전트에게 제공.
- 예시: "이 500 에러는 지난주 배포된 Redis 분산 락 로직의 타임아웃 설정 오류와 유사합니다." 같은 구체적인 답변이 가능.

## 3. 에이전트의 능동적 조사 (Tool Calling)
- 로그 정보만으로 원인이 불충분할 때, 에이전트가 스스로 정보를 더 수집하는 단계가 필요.
- 인프라 상태 조회 도구: 에이전트가 AWS CloudWatch나 Kubernetes API를 호출해 "현재 DB CPU 부하가 90% 이상인가?", "네트워크 지연이 발생 중인가?"를 직접 확인.
- 코드 레포지토리 조회: 에러 스택 트레이스(Stack Trace)가 찍혔을 때, 에이전트가 실제 소스 코드의 해당 라인을 조회하여 로직 오류 가능성을 분석.

## 4. 슬랙 알림의 '등급화' 및 '피드백' 루프
모든 오류를 슬랙으로 보내면 '알림 피로도' 때문에 중요한 메시지를 놓침.
- 심각도 분류 (Severity): 400 에러(사용자 실수 등)는 요약해서 일별 리포트로 보내고, 500 에러나 결제 중복 같은 크리티컬한 사안은 즉시 태깅(@Admin)을 포함해 전송하는 로직이 필요.
- (참고) 피드백 버튼: 슬랙 메시지에 [해결됨], [오탐지], [추가 조사 필요] 같은 버튼을 달아두세요. 관리자가 누른 피드백을 에이전트가 학습하여 나중에 더 정확한 해결법을 제시.

## 5. 보충해야 할 핵심 프로세스: '실행 방지 가드레일'
운영 에이전트에서 가장 조심해야 할 부분입니다.
- 만약 에이전트가 해결책으로 "파드 재시작"이나 "블랙리스트 등록"을 제안할 때, 직접 실행하게 할 것인지 아니면 사람의 승인을 거치게 할 것인지 결정해야 합니다. 초기에는 슬랙 메시지에 [실행 승인] 버튼을 두어 사람이 최종 컨펌을 하는 구조를 강력히 추천.
